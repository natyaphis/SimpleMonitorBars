SimpleMonitorBars Plan 150

Topic
Split the current duration bar into two independent bar types:
- aura
- cooldown

The goal is to improve stability and clarity without polluting unrelated features or requiring a large settings UI rewrite.


1. Overall Plan

Current problem:
The existing duration bar mixes two responsibilities:
- aura duration tracking
- spell cooldown timing

This creates unstable runtime behavior because the bar may depend on aura APIs, auraInstanceID, cooldown viewer mappings, and combat-sensitive cache refresh paths at the same time. The Shield Block issue is a direct example of this mixed architecture.

Target architecture:
- stack: stack count tracking
- charge: charge count and recharge tracking
- aura: aura duration tracking only
- cooldown: spell cooldown tracking only

Key principles:
- one bar type should have one primary data source
- aura bars must not rely on cooldown viewer data as their main runtime source
- cooldown bars must not rely on aura instance data as their main runtime source
- existing settings should be reused wherever possible
- old profiles should keep working
- the first phase should prefer compatibility and low risk over aggressive cleanup


2. Design Goals

- stable in and out of combat
- simple runtime responsibilities
- minimal UI change
- minimal config migration risk
- no regression to stack and charge bars
- easy to debug and extend later


3. New Bar Type Responsibilities

Aura bar:
- tracks the remaining duration of a helpful or harmful aura on a unit
- primary data source is aura data
- supports player and target tracking
- supports reverse growth, text, colors, visibility rules, border and style options

Cooldown bar:
- tracks spell cooldown remaining time
- primary data source is spell cooldown APIs
- does not depend on unit aura state
- supports text, colors, reverse growth, visibility rules, border and style options

Charge bar remains separate:
- charge count and recharge behavior stay in the existing charge implementation


4. Runtime Architecture Plan

4.1 Dispatcher

UpdateAllBars should eventually dispatch by these effective runtime types:
- stack
- charge
- aura
- cooldown
- duration -> aura compatibility mapping

This means old duration configs still behave, but internally they are treated as aura bars.


4.2 Aura Bar Architecture

Aura bars should use a pure aura pipeline:
- resolve aura state
- render aura state

Recommended internal state fields:
- active
- unit
- spellID
- auraInstanceID
- duration
- expirationTime
- remaining
- sourceKind

Recommended source priority:
1. C_UnitAuras.GetPlayerAuraBySpellID / GetAuraDataByIndex / GetAuraDataByAuraInstanceID
2. AuraUtil iteration
3. UnitAura fallback

Important rule:
If aura duration can be determined from duration and expirationTime, the bar must render even if auraInstanceID is not available.

This removes the current failure mode where the addon knows the aura exists but cannot animate the bar.


4.3 Cooldown Bar Architecture

Cooldown bars should use a pure cooldown pipeline:
- resolve cooldown state
- render cooldown state

Recommended internal state fields:
- active
- spellID
- durationObject
- remaining
- isOnCooldown
- sourceKind

Primary source priority:
1. direct spell cooldown APIs
2. existing cooldown viewer mapping only as enhancement or alias support

Cooldown bars should not look up unit aura state.


4.4 Event Routing

UNIT_AURA:
- update aura bars
- update stack bars
- do not update cooldown bars

SPELL_UPDATE_COOLDOWN:
- update cooldown bars
- keep existing charge logic behavior

SPELL_UPDATE_CHARGES:
- update charge bars only

PLAYER_REGEN_ENABLED:
- do not use combat leave as a required recovery path for aura bars
- cooldown viewer rescans, if still needed, should only benefit cooldown and charge related paths

This isolates combat-sensitive behavior instead of letting it leak into aura duration logic.


5. Settings UI Plan

The settings UI should be changed minimally.

Recommended user-facing bar type list:
- Stack
- Charge
- Aura
- Cooldown

Compatibility behavior:
- existing duration bars should appear as Aura when edited

Shared settings reused for both Aura and Cooldown:
- spell selection
- size
- texture
- colors
- text display
- border and mask style
- orientation
- reverse growth
- visibility condition

Type-specific visibility:
- Aura shows unit selection
- Cooldown hides unit selection
- Charge keeps charge-specific options
- Stack keeps stack-specific options

The layout should not be redesigned. Only bar-type options and conditional field visibility should be adjusted.


6. Improvement Boundaries

This plan should not:
- redesign the full settings page
- rewrite stack behavior
- rewrite charge behavior
- remove old duration support immediately
- require a large profile migration in phase one

This plan should:
- isolate aura and cooldown responsibilities
- make aura runtime independent from cooldown viewer rescans
- keep profile compatibility


7. Module Change List

Primary modules to change:

1. Bars/Bars.lua
- introduce new runtime update paths for aura and cooldown bars
- replace current duration runtime logic with compatibility forwarding
- isolate rendering logic
- isolate resolve logic
- narrow event reactions by bar type

2. Skins/MonitorTab.lua
- update bar type options shown in the editor
- map legacy duration entries to Aura in the UI
- conditionally show unit controls only for aura bars
- keep shared controls unchanged

3. Core/defaults or schema-related files if needed
- only minimal changes if bar type enumeration or saved defaults need extension
- avoid forced migration in the first pass

4. Bars/Bridge.lua
- keep existing cooldown viewer infrastructure
- stop using it as a runtime dependency for aura bars
- retain it only for cooldown or charge related enhancement paths

Modules that should remain functionally untouched:
- stack runtime logic
- charge runtime logic except shared dispatch integration
- most of the settings layout and style controls


8. Function-Level Refactor Draft

Recommended new or refactored functions in Bars/Bars.lua:

Aura side:
- GetEffectiveBarType(cfg)
  Returns aura for legacy duration bars.

- ResolveAuraBarState(barFrame)
  Builds the normalized aura runtime state.

- FindAuraStateBySpellID(unit, spellID)
  Wraps the multi-layer aura lookup and returns normalized aura data.

- NormalizeLegacyAuraData(rawAura, unit, spellID, sourceKind)
  Converts API-specific aura results into a shared shape.

- ApplyAuraDurationFallback(seg, state, cfg, textRegion)
  Manually drives the status bar from duration and expirationTime when no duration object exists.

- RenderAuraBar(barFrame, state)
  Applies progress, text, color, active state, and empty state.

Cooldown side:
- ResolveCooldownBarState(barFrame)
  Builds normalized cooldown runtime state.

- RenderCooldownBar(barFrame, state)
  Applies cooldown progress and text display.

- UpdateCooldownOnlyBar(barFrame)
  Runtime entry point for cooldown bar type.

Compatibility:
- UpdateAuraBar(barFrame)
  Runtime entry point for aura bar type.

- UpdateDurationBar(barFrame)
  Temporary compatibility wrapper that forwards to UpdateAuraBar.

- UpdateBarByType(barFrame)
  Internal dispatch helper if needed.

Event helpers:
- ShouldHandleAuraEventForBar(barFrame, unit)
- ShouldHandleCooldownEventForBar(barFrame)

Settings side:
- GetDisplayBarType(savedBarType)
  Returns aura when saved type is duration.

- GetPersistedBarType(selectedType)
  Allows phased migration strategy if needed.


9. Configuration Compatibility Strategy

Primary compatibility rule:
- saved barType = duration should continue to work automatically

Recommended first-phase strategy:
- runtime interprets duration as aura
- settings UI displays duration entries as Aura
- no forced bulk migration on load

Benefits:
- lower risk
- easier rollback
- old profiles continue to function

Optional later migration:
- once aura bars are stable, add a schema update that rewrites duration -> aura
- this should happen only after runtime behavior is verified

Compatibility notes:
- all shared fields should be preserved as-is
- unit remains meaningful for aura
- unit should be ignored or hidden for cooldown
- no profile reset should be required


10. Phase-Based Commit Plan

Phase 1: Runtime split groundwork
Scope:
- add effective bar type compatibility handling
- introduce new aura and cooldown runtime entry points
- keep old duration path as wrapper to aura

Expected result:
- no user-facing breakage
- no settings page breakage yet

Suggested commit theme:
- "Refactor runtime bar type dispatch for aura and cooldown split"


Phase 2: Aura bar stabilization
Scope:
- implement pure aura resolution path
- remove cooldown viewer dependency from aura runtime
- add legacy duration fallback using duration and expirationTime
- make aura behavior stable in combat

Expected result:
- fixes Shield Block style issues
- aura bars behave consistently in combat and out of combat

Suggested commit theme:
- "Implement pure aura duration bar runtime with fallback timing"


Phase 3: Cooldown bar extraction
Scope:
- implement cooldown-only runtime
- separate it cleanly from aura logic
- keep charge-specific behavior separate

Expected result:
- cooldown bars use a dedicated path
- mixed duration semantics are gone internally

Suggested commit theme:
- "Add dedicated cooldown bar runtime"


Phase 4: Settings UI minimal update
Scope:
- expose Aura and Cooldown as selectable types
- map legacy duration to Aura in editor display
- conditionally show relevant fields

Expected result:
- users can create the new bar types directly
- existing profiles remain editable

Suggested commit theme:
- "Update settings UI for aura and cooldown bar types"


Phase 5: Compatibility cleanup
Scope:
- optional schema migration duration -> aura
- remove dead code only after verification
- keep compatibility if any uncertainty remains

Expected result:
- cleaner saved data model
- reduced maintenance burden

Suggested commit theme:
- "Normalize legacy duration bars to aura type"


11. Validation Checklist

Aura bar validation:
- player aura updates correctly in combat
- target aura updates correctly in combat
- no empty-progress bug when aura exists but auraInstanceID is missing
- progress and text remain continuous across combat state changes

Cooldown bar validation:
- cooldown starts and expires correctly
- text matches cooldown remaining time
- GCD is not treated as the main tracked cooldown

Compatibility validation:
- old duration bars load and behave as aura bars
- old profiles require no manual repair
- editing a legacy bar in the UI does not lose settings

Regression validation:
- stack bars unchanged
- charge bars unchanged
- shared styling and visibility behavior unchanged


12. Final Recommendation

The safest path is not to heavily redesign the addon all at once.

The safest path is:
- split runtime responsibilities first
- keep old duration configs compatible
- make Aura the stabilized replacement for old duration behavior
- add Cooldown as a distinct bar type with minimal UI change

This keeps the architecture clean while limiting risk to the existing addon behavior.
