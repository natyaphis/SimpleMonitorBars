SimpleMonitorBars Skillset / Default Value Analysis

Purpose
- This file records where addable monitor entries come from and what default values are appropriate.
- It is meant as analysis data for future per-skill / per-buff default overrides.
- It is not runtime data and should not be packaged with the addon.

Packaging Note
- Asset is currently excluded from release packaging.
- Source: .github/workflows/package-addon.yml
- The package workflow includes: --exclude "Asset/"

Where Addable Entries Come From
1. Blizzard cooldown viewer catalog
- Source: Skins/MonitorTab.lua -> ShowCatalog()
- SMB calls MB:GetSpellCatalog()
- MB:GetSpellCatalog() is built from Blizzard cooldown viewer frames in Bars/Bridge.lua
- This means most addable entries are runtime-driven and are not statically listed anywhere in the repository.

2. Explicit special entries injected by SMB
- Current explicit special case:
  - Icicles / 小冰刺
  - Source: Skins/MonitorTab.lua
  - Injected only when it does not already appear in the Blizzard viewer catalog

Current Default Value Source
- New entries are created by NewBarDefaults() in Skins/MonitorTab.lua
- Current generic defaults relevant to trigger settings:
  - unit = player
  - maxStacks = 5
  - maxCharges = 0
  - isChargeSpell = (barType == charge)
- AddBar() only applies extra default logic for charge bars:
  - maxCharges is auto-filled from C_Spell.GetSpellCharges()
- There is currently no equivalent per-spell logic for:
  - stack bars needing target instead of player
  - stack bars needing maxStacks other than 5
  - duration bars needing target instead of player

Why The "Shatter / 碎裂" Problem Happens
- The problem does not appear to be caused by a special bug in "Shatter" itself.
- It happens because SMB does not have a per-spell default override table.
- If a newly added stack aura should default to:
  - unit = target
  - maxStacks = 20
  then SMB currently cannot infer that unless the catalog entry already carries the correct unit and the user manually fixes maxStacks.

Known Entry Types And Recommended Defaults

A. Charge / Cooldown skills
- Typical source: Blizzard cooldown viewer cooldown catalog
- Recommended defaults:
  - barType = charge
  - unit = player
  - maxCharges = auto from C_Spell.GetSpellCharges()
  - showCondition = always
- Current SMB handling:
  - Good enough by default
  - Already has auto maxCharges logic

B. Player buff stacks
- Typical source: Blizzard aura viewer entries for helpful auras on player
- Recommended defaults:
  - barType = stack
  - unit = player
  - maxStacks = spell-specific override if known, otherwise 5 as generic fallback
- Current SMB handling:
  - unit usually acceptable if the catalog entry unit is player
  - maxStacks remains too generic

C. Target debuff stacks
- Typical source: auras that conceptually belong on target
- Recommended defaults:
  - barType = stack
  - unit = target
  - maxStacks = spell-specific override
- Current SMB handling:
  - vulnerable to wrong defaults because generic fallback is unit=player, maxStacks=5

D. Duration buffs / debuffs
- Recommended defaults:
  - barType = duration only when the tracked effect is mainly time-oriented rather than stack-oriented
  - unit should follow the actual aura owner:
    - player for self buffs
    - target for target debuffs
- Current SMB handling:
  - still generic unless the catalog entry carries the correct unit

Known / High-Confidence Special Cases

1. Icicles / 小冰刺
- Confidence: high
- Reason: explicitly hardcoded in SMB as a special injected catalog entry
- Recommended defaults:
  - barType = stack
  - unit = player
  - maxStacks = 5
- Current SMB state:
  - unit is already explicitly set to player in specialEntries
  - maxStacks still comes from generic defaults, but generic 5 happens to be correct here

2. Shatter / 碎裂
- Confidence: medium
- Reported desired defaults from user testing:
  - barType = stack
  - unit = target
  - maxStacks = 20
- Current SMB state:
  - no explicit override found in repository
  - falls back to generic defaults unless catalog entry unit happens to be target
- Conclusion:
  - this is a real candidate for a dedicated override rule

Other Entries Likely To Have The Same Problem
- Any stack-based target debuff
- Any stack-based aura whose logical cap is not 5
- Any manually added buff / aura that is not charge-based and needs spell-specific defaults

What Cannot Be Fully Enumerated Statically
- SMB does not ship a fixed spell list for all addable skills and buffs.
- The majority of entries come from Blizzard cooldown viewer state at runtime.
- Because of that, the repository alone cannot provide a complete static list of every addable skill or buff across all classes/specs.

Recommended Future Override Strategy
- Add a per-spell defaults table keyed by spellID.
- Support override fields:
  - defaultBarType
  - defaultUnit
  - defaultMaxStacks
  - defaultMaxCharges
  - defaultShowCondition
- Apply these overrides in AddBar() after NewBarDefaults() and before inserting into cfg.bars.

Suggested First Overrides To Add Later
- Icicles / 小冰刺:
  - unit = player
  - maxStacks = 5
- Shatter / 碎裂:
  - unit = target
  - maxStacks = 20

Summary
- The incorrect defaults are caused by generic NewBarDefaults(), not by a unique runtime bug in one spell.
- Charge skills already have partial auto-default logic.
- Stack / duration auras are the weak area and need a spell-specific override table.
- Asset is currently excluded from release packaging, so this analysis file is safe to keep there.
